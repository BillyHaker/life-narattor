## Goal
Restore production voice transcription and fix stale UI state after transcription retry failures.

## Plan
1. Add backend support for Volcengine/Doubao `X-Api-Key` transcription authentication.
2. Keep legacy `X-Api-App-Key` + `X-Api-Access-Key` authentication for older console credentials.
3. Add a server-side OpenAI transcription fallback when Doubao fails and OpenAI credentials are present.
4. Fix iOS transcription task cleanup so feed cards refresh after a task exits.
5. Make failed transcription status take precedence over stale active-task UI labels.
6. Verify server syntax, iOS build, tests, and transcription smoke behavior.

## Detection Plan
- Expected behavior: online `/v1/transcribe` should not fail solely because the server only supports old Doubao credentials.
- Expected behavior: if a transcription retry fails, both detail sheet and feed card show failed/offline status instead of stale `正在转写…`.
- Pass criteria: server syntax passes; iOS build passes; tests pass; local smoke test reaches a real provider path or returns a clear provider/config error.
- Failure signal: feed remains `正在转写…` after the detail sheet shows failure; server returns unclear `server_error` for known transcription configuration/auth errors.
- Regression surface: recording, retry transcription, backend `/v1/transcribe`, usage quota accounting, record feed card status display.

## Unexpected Behavior Diagnosis
**Phenomenon:** App Store build shows `AI 配置缺失，无法拆分。` or failed transcription, and retry leaves the feed card showing `正在转写…`.

**Reproduction:** POSTing a small audio multipart request to the production backend `/v1/transcribe` returned HTTP 500 with a Doubao provider 401 payload: `requested grant not found in SaaS storage`.

**Implementation check:**
- Backend `server/server.js` only supported the legacy Doubao app/access-key credential headers and surfaced raw provider failures as generic server errors.
- iOS `CaptureFeedViewModel` removed the async task from `transcriptionTasks` only after the failure branch refreshed the feed. The feed therefore rebuilt while the task was still marked active.
- UI components used `isTranscriptionActive` directly, so a failed/offline voice capture could still render as `正在转写…`.

**Spec/product check:** Voice transcription should be resilient and should never leave the user in an indefinite in-progress state. The feature may fail due to provider credentials, but the UI must report failure and allow retry.

**Root cause:** Backend provider auth/config mismatch plus client-side stale active-task state ordering.

**Fix:** Add Doubao API-key credential support, fallback to OpenAI audio when Doubao config/auth fails and OpenAI is configured, sanitize known transcription provider errors, and refresh iOS feed state after clearing task tokens.

## Implementation
- Backend transcription now accepts preferred `DOUBAO_API_KEY` credentials and falls back to legacy `DOUBAO_APP_ID` + `DOUBAO_ACCESS_TOKEN` when no API key is set.
- Backend uses `DOUBAO_USER_ID` for Doubao ASR `user.uid`, defaulting to `life-narrator`.
- Backend falls back from Doubao to OpenAI audio transcription for known provider config/auth/format failures when `OPENAI_API_KEY` is available.
- Backend maps known transcription provider failures to stable 503 response codes instead of exposing raw provider errors as general failures.
- iOS transcription queue now tags each queued transcription task with a token, clears only the matching active task, refreshes feed data after clearing, and posts a capture-state notification.
- iOS feed/detail UI now shows `正在转写…` only when a voice capture is both active and still pending; failed/offline states can show retry immediately.

## Operational notes
- The currently deployed Render backend was observed returning Doubao 401 before this change is deployed.
- To fix production immediately after deploy, either configure a valid `DOUBAO_API_KEY`/Doubao resource grant or ensure `OPENAI_API_KEY` is configured so fallback transcription can run.
- Existing App Store builds can benefit from backend fallback after deploy because they call the same backend URL. The stale feed UI fix requires a new app binary release.
