## 非预期行为诊断

**现象**：有网络、聊天正常、转写正常，但详情页“拆分”持续显示“当前未拆分；网络可用后会自动拆分”。
**复现步骤**：打开未拆分记录详情 -> 点击“重新拆分” -> 页面仍回到同一文案。

**第一层检查（实现）**：
- `CaptureFeedViewModel.performAtomization(for:)` 会把 `invalidResponse` / `emptyResponse` / 部分 HTTP 错误写回 `pendingSplit`。
- `CaptureDetailSheet.emptyAtomStateText` 对 `pendingSplit` 固定显示“网络可用后会自动拆分”。
- `server/server.js` 的 `/v1/atomize` schema 对 `type` 约束过松，存在返回不可解码值的风险。

**第二层检查（规范）**：
- “无网先记录、恢复后自动拆分、可手动重新拆分”的方向合理。
- 规范没有要求把所有失败都伪装成网络问题，当前实现偏离了预期。

**第三层检查（产品）**：
- 产品方向正确，问题不在产品需求，而在实现层状态机和错误映射。

**根本原因判断**：实现层问题。atomize 真失败被错误回写为 `pendingSplit`，再被 UI 误翻译成“网络可用后再拆分”；同时 backend atomize 契约过松，增加了解码失败概率。

**建议修复方案**：
- 新增 `atomizationError` 持久化字段。
- 仅真实可延期错误保留 `pendingSplit`；`invalidResponse` / `emptyResponse` 改为 `splitFailed`。
- 详情页展示真实 `atomizationError`。
- 收紧 backend `/v1/atomize` 的 `type` 枚举。

## Work Log

- `BackendAIService.post(...)` 的解码失败统一映射为 `AIServiceError.invalidResponse`。
