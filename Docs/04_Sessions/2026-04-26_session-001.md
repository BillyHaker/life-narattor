# Session 001 — Timeline Snapshot Summaries Phase 1

## Goal
Move Timeline summary cards away from real-time ad-hoc narration and toward a daily-refreshed snapshot model: yesterday, past 7 days, and past 30 days.

## Plan
- Define timeline snapshot models and a snapshot persistence/refresh service.
- Wire Timeline summary cards to snapshots instead of realtime stats-only copy.
- Change summary semantics to `昨日 / 过去7天 / 过去30天` while leaving day cards as the live browse surface.
- Use foreground/on-open repair instead of relying on an exact iOS background execution time.
- Verify with build, tests, and direct copy wiring checks.

## Detection Plan
- Expected behavior:
  - `今天` tab shows today's day nodes, but the summary references yesterday.
  - `本周` and `本月/近30天` read from snapshot storylines instead of realtime count-only blurbs.
  - Snapshots persist and do not require realtime regeneration on every render.
- Detection path:
  - Inspect `TimelineScreen` and `TimelineReviewSnapshotService`.
  - Build the app and run unit tests.
  - Open Timeline and switch `今天 / 本周 / 本月 / 近30天`.
- Pass criteria:
  - Build/tests pass.
  - Snapshot labels appear as `昨日故事线 / 过去 7 天故事线 / 过去 30 天故事线`.
  - Summary cards render snapshot text when available and graceful fallback copy when not.
- Failure signals:
  - Timeline still shows only count-based summary text.
  - Snapshot refresh blocks page rendering or fails compilation due to actor/async issues.
  - No persisted snapshot is loaded on reopen.
- Regression surface:
  - Timeline summary card
  - Timeline open/foreground lifecycle
  - Review retrieval service reuse
  - ArtifactEntity JSON persistence

## Work Log
- Audited current Timeline page and confirmed the user concern was specifically about the Timeline summary card, not the dedicated AI review pages.
- Chose a first-phase architecture that stores summary snapshots in `ArtifactEntity` rather than adding a new schema.
- Added `TimelineReviewSnapshotKind` and `TimelineReviewSnapshotPayload` to model `昨日 / 过去7天 / 过去30天` review snapshots.
- Added `TimelineReviewSnapshotService` to:
  - load persisted snapshots
  - detect when a snapshot is stale by comparing its stored range to the current desired range
  - regenerate snapshots from `ReviewRetrievalService` plus AI analysis
  - persist fallback summaries when AI is unavailable or the range is empty
- Wired `TimelineScreen` to load snapshots on appear, refresh them when the app becomes active, and render snapshot-first copy in the summary card.
- Changed Timeline summary semantics:
  - `今天` -> `昨日故事线`
  - `本周` -> `过去 7 天故事线`
  - `本月/近30天` -> `过去 30 天故事线`
- Updated Timeline header copy so the page explains that raw day nodes stay live while the summary is based on a previously completed period.
- Preserved fallback behavior: if a snapshot has not yet been regenerated, the page still renders calm placeholder copy and current day-count context.

## 非预期行为诊断

**现象**：用户希望 Timeline 的每日/周/月总结不要依赖实时更新，而是基于已经结束的时间段给出稳定参考。当前页面 summary 只是统计提示，且会让人误解成“今天/本周正在实时被理解”。
**复现步骤**：打开 Timeline 的 `今天 / 本周 / 本月`，summary 区只显示 `有几天 / 几条片段` 一类说明。

**第一层检查（实现）**：
- `TimelineScreen` 的 `rangeSummaryText` 完全由 `days.count` 和 `recordCount` 拼接。
- 没有任何持久化 summary；页面每次只是在本地解释当前列表。

**第二层检查（规范）**：
- 时间线页的职责应该是“按时间回看节点”，不是“实时现算 AI 总结”。
- 当前实现没有违反已有 retrieval 规范，但它没有建立稳定的 period summary 契约。

**第三层检查（产品）**：
- 用户更需要“回看参考”而不是“实时解释”。
- 用已结束时间段生成快照，比实时总结更符合低压力和稳定预期。

**根本原因判断**：产品表达和实现策略同时偏向实时列表解释，没有把“周期性回顾快照”作为单独的一层建立起来。
**建议修复方案**：
- 让时间线保留实时节点浏览。
- 让 summary 改为 `昨日 / 过去7天 / 过去30天` 快照。
- 用前台补跑和失效检查保证快照更新，而不是依赖 iOS 精确定时后台运行。

# Session 002 — Timeline Snapshot Freshness Fix

## Goal
Remove the distracting snapshot generation-time chip from Timeline summary cards and fix the bug where an empty snapshot could be incorrectly reused after new records had been added later in the same covered period.

## Plan
- Remove the `整理于 ...` chip from Timeline summary UI.
- Extend snapshot freshness checks so validity depends on range data, not just date-range boundaries.
- Ensure an empty snapshot is invalidated as soon as the covered range now contains formal records.
- Verify with build and unit tests.

## Detection Plan
- Expected behavior:
  - Timeline summary cards no longer show `整理于 ...`.
  - A previously empty snapshot should refresh once new formal records appear in the same covered period.
- Detection path:
  - let Timeline cache an empty summary
  - add a new formal record that falls into that summary range
  - reopen Timeline or foreground the app
- Pass criteria:
  - the card no longer reports `0 天有记录 / 0 条材料` when qualifying records now exist
  - the `整理于 ...` chip is gone
- Failure signals:
  - empty snapshot remains visible after new records are added
  - snapshot validity still depends only on `rangeStart/rangeEnd`
- Regression surface:
  - Timeline summary card
  - snapshot persistence payload shape
  - foreground refresh behavior

## Work Log
- Audited the user report showing `过去 30 天故事线` with `0 天有记录 / 0 条材料` despite real records existing.
- Reconfirmed the earlier snapshot architecture and traced the specific bug to `TimelineReviewSnapshotService.isCurrent`.
- Verified that phase-1 freshness logic only compared `rangeStart` and `rangeEnd`, which let a same-day empty snapshot remain “current” even after later data changes.
- Removed the `整理于 ...` chip from Timeline summary cards because it amplified user confusion without helping recovery.
- Extended `TimelineReviewSnapshotPayload` with `latestRecordAt` so the snapshot can remember the data state it was built from.
- Added `currentFreshness(in:)` to compute three lightweight freshness signals for eligible Timeline records:
  - active day count
  - total formal-record count
  - latest formal-record timestamp
- Updated snapshot validity checks so Timeline now refreshes when any of those signals change.
- Added a special guard to ensure an old empty snapshot is not reused once the covered range now has qualifying records.

## 非预期行为诊断

**现象**：Timeline 会显示 `过去 30 天故事线`，但内容仍然是 `0 天有记录 / 0 条材料`，即使用户已经在这个范围内留下正式记录。
**复现步骤**：先在某个时间点生成一个空的 Timeline snapshot；随后在同一自然日里新增正式记录；重新打开 Timeline。

**第一层检查（实现）**：
- Snapshot 读取逻辑优先复用已保存的 `ArtifactEntity`。
- `isCurrent` 只比较 `rangeStart` 和 `rangeEnd`。
- 只要范围边界没变，旧 snapshot 就继续被视为有效。

**第二层检查（规范）**：
- Snapshot summary 的目标是给用户提供结束时段的参考，而不是执着于“是否刚刚生成”。
- 但稳定不等于忽略数据变化；只用范围判断 freshness 会破坏 summary 的可信度。

**第三层检查（产品）**：
- `整理于 4月26日 15:53` 这样的标签会把系统内部状态直接暴露给用户。
- 当 summary 本身又明显错误时，这个时间标签会进一步放大不信任感。

**根本原因判断**：问题主要不在 AI 分析或 retrieval 本身，而在 snapshot freshness 失效判断过粗，导致空 snapshot 被错误复用。
**建议修复方案**：
- 去掉 `整理于 ...` 标签。
- 用范围内正式记录的数据变化信号参与 freshness 判断。
- 只要范围内有新增正式记录，就让 snapshot 在前台补跑时重新生成。

# Session 003 — Timeline Snapshot Overview Plan Fix

## Goal
Fix the remaining Timeline snapshot summary issue where the period had visible formal records and atomization payloads, but the snapshot card still failed to produce useful summary data.

## Plan
- Separate system-generated range summaries from user natural-language retrieval planning.
- Add a dedicated overview retrieval plan for fixed windows such as `过去 7 天` and `过去 30 天`.
- Keep ordinary AI search on the existing natural-language query builder.
- Make snapshot fallback stats factually reflect records seen by freshness checks.
- Add regression tests for the retrieval plan classification.

## Detection Plan
- Expected behavior:
  - `过去 7 天` and `过去 30 天` programmatic summaries use `mode == .overview`.
  - Timeline snapshot cards do not show zero stats when the covered range has formal records.
- Detection path:
  - inspect the current simulator database records and payloads
  - run build and unit tests
  - check the new `RetrievalPlanBuilderTests`
- Pass criteria:
  - build/tests pass
  - new tests prove spaced numeric labels remain overview plans
  - fallback payloads use real freshness counts when summary data is unavailable
- Failure signals:
  - range summaries are still routed through focused/comparison retrieval
  - card stats disagree with visible Timeline day cards
- Regression surface:
  - Timeline snapshots
  - weekly/monthly range review pages
  - ordinary user natural-language search

## Work Log
- Rechecked the real simulator database for the user-visible `4/21` and `4/24` records.
- Confirmed those captures are formal `log` records with `processingState == atomsReady` and existing `atomization_payload.record_units`.
- Confirmed the saved `last7Days` and `last30Days` snapshots had `latestRecordAt` populated while still reporting `activeDayCount = 0` and `totalRecordCount = 0`.
- Identified the deeper issue: system snapshots were using the same natural-language query inference path as user searches.
- Added `makeSystemOverviewPlan(periodLabel:range:)` so fixed-window summaries bypass query parsing and always use overview retrieval.
- Routed `makeOpenReviewPlan(periodLabel:range:)` through the new system overview plan.
- Updated Timeline snapshot fallback generation to preserve real freshness counts even if range review data is still unavailable.
- Added `RetrievalPlanBuilderTests` for spaced labels like `过去 7 天` and `过去 30 天`.

## 非预期行为诊断

**现象**：Timeline 的 `过去 7 天故事线` 显示没有有效总结，甚至显示 `0 天有记录 / 0 条材料`，但下方同一范围内有可见日期卡片。
**复现步骤**：当前模拟器数据库中，打开 Timeline `本周`；下方能看到 4/21、4/24 的正式记录，但上方 snapshot summary 仍为空。

**第一层检查（实现）**：
- `TimelineScreen` day cards 直接读取正式 `CaptureEntity`，所以能显示记录。
- `TimelineReviewSnapshotService` 的 freshness 检查也能看到范围内最新正式记录。
- `ReviewRetrievalService.makeRangeReviewData` 依赖 `makeOpenReviewPlan` 生成 retrieval plan。
- `makeOpenReviewPlan` 原本通过自然语言 query `我过去 7 天都干了什么，有什么主要变化` 推断 plan。
- `RetrievalPlanBuilder` 不识别带空格数字的 `过去 7 天` 为 broad review window，且 `变化` 可能使 query 走向 focused/comparison。
- focused plan 没有匹配 filter 时，会过滤掉 otherwise valid overview units。

**第二层检查（规范）**：
- 用户自然语言检索应该保留 query inference，因为它需要理解用户提问。
- 系统固定周期总结不应该依赖自然语言推断；它的语义已经由调用方确定。

**第三层检查（产品）**：
- Timeline snapshot 是“稳定周期回顾”，不是用户即时搜索。
- 它应该优先保证覆盖范围和事实一致性，再尝试 AI 叙事。

**根本原因判断**：系统周期总结和用户自然语言搜索共用了同一套 query parser，导致固定窗口总结被错误归类为 focused/comparison 检索。
**建议修复方案**：
- 给系统周期总结单独建立 overview retrieval plan。
- 保留用户搜索的自然语言解析路径。
- fallback 时显示真实记录统计，不再把 summary 失败说成没有记录。

# Session 004 — Timeline Stale Snapshot Display Guard

## Goal
Prevent Timeline from presenting known-stale cached snapshots as current summary facts, especially old empty snapshots that disagree with visible day cards.

## Plan
- Let Timeline validate cached snapshots before inserting them into UI state.
- Hide stale cached snapshots and show refresh/fallback state instead.
- Refresh the selected scope first.
- Update snapshot UI incrementally as each kind finishes.
- Verify with build and unit tests.

## Detection Plan
- Expected behavior:
  - stale cached snapshots should not render as final facts
  - current tab refresh should not wait for unrelated snapshot windows
- Detection path:
  - use an old empty `last7Days` snapshot with existing formal records
  - open Timeline `本周`
  - observe summary card before and after refresh
- Pass criteria:
  - stale `0 天 / 0 条` snapshot is hidden
  - current `本周` snapshot refreshes first and updates as soon as it is available
- Failure signals:
  - old cached result appears as normal summary copy
  - UI updates only after all snapshot kinds finish
- Regression surface:
  - Timeline `onAppear` refresh
  - scope switching
  - foreground refresh

## Work Log
- Rechecked the simulator database after the previous fix and confirmed `last7Days` and `last30Days` snapshots were now generated successfully.
- Identified the remaining user-facing issue as display order: Timeline loaded cached snapshots into UI before validating whether they were current.
- Added `TimelineReviewSnapshotService.isSnapshotCurrent(_:)` to expose the same freshness contract used by `refreshIfNeeded`.
- Updated Timeline loading so only current cached snapshots are inserted into `snapshots`.
- Made stale cached snapshots clear from UI state, allowing the card to show loading/fallback while refresh runs.
- Changed refresh ordering so the selected snapshot kind runs first.
- Changed refresh updates to write each snapshot into state immediately after it finishes.

## 非预期行为诊断

**现象**：用户仍然看到 `过去 7 天故事线` 的旧空内容，但数据库里已经有有效 `last7Days` snapshot。
**复现步骤**：打开 Timeline `本周`，页面先读取本地 cached snapshot，再异步刷新。

**第一层检查（实现）**：
- `TimelineScreen.refreshSnapshotsIfNeeded` previously loaded all cached snapshots unconditionally.
- It refreshed all snapshot kinds and updated UI only once at the end.
- A stale cached snapshot could therefore be visible long enough to look like the real result.

**第二层检查（规范）**：
- Snapshot cache is a performance and stability layer, not a source of truth when freshness says it is stale.
- Stale snapshots should not be displayed as final user-facing facts.

**第三层检查（产品）**：
- Showing an old empty result damages trust more than showing a short refresh state.
- The selected tab should feel responsive before background maintenance of other ranges.

**根本原因判断**：refresh generation was fixed, but Timeline still trusted stale cached display state too early.
**建议修复方案**：
- Validate cached snapshots before showing them.
- Refresh selected scope first.
- Update UI incrementally as each refreshed snapshot arrives.
