# Session 001 — Timeline Memory Spine Pass 1

## Goal
Reshape the Timeline home screen so it feels like a calm date-based memory browser instead of a grouped record list with synthetic narrative states.

## Plan
- Replace `TimelineDay`'s old `highlights + hasNarrative` shape with fields that describe a real day card.
- Redesign the Timeline home header, range summary, empty state, and day cards around low-pressure review.
- Remove fake narrative state inference based on record counts.
- Sync Timeline day construction in retrieval/search/review entry points.
- Verify with build, test, and copy scan.

## Detection Plan
- Expected behavior: Timeline home reads like a date-based review surface, not a `生成/查看日记` tool panel.
- Detection path: inspect `TimelineScreen`/related builders, run copy scan for old timeline phrases, build the app, run tests.
- Pass criteria: build and tests pass; timeline cards show `primaryLine + secondaryLines + 回看这一天`; no remaining `生成日记/整理成今日叙事/查看今日叙事/结构化片段` copy in screens/data/models.
- Failure signals: compile errors from the model migration, cards still showing fake narrative states, or engineering-language empty/error copy surfacing in timeline flow.
- Regression surface: Timeline home, weekly/monthly day previews, search-to-day navigation, retrieval-built day cards.

## 非预期行为诊断

**现象**：某些日期在 Day Detail 中能显示“今日叙事”或原始记录，但 `AI 轻回应` 没有内容，之前还会出现“这一天还不适合给出稳定的 AI 回应”一类提示。  
**复现步骤**：在时间线打开某一天，看到当天确实有记录，但 AI 回顾区域为空或之前显示材料不足提示。

**第一层检查（实现）**：
- `DayDetailScreen` 只有在 `NarrativeMaterial.representativeUnits` 非空时才会请求 AI。
- `MemoryIndexStore` 只从 `atomization_payload.recordUnits` 读取回顾材料。
- `CaptureFeedViewModel` 的自动拆分队列此前只会捞 `processingState == .pendingSplit` 的记录。
- 助手语音 capture 会显式停在 `.cleanReady`，旧的助手归档路径还能生成 atoms 但不写 `atomization_payload`。

**第二层检查（规范）**：
- 规范方向是“只有正式记录进入回顾，AI 基于统一拆分材料工作”。
- 现状与这个方向不一致：时间线/Day Detail/检索入口没有严格区分正式记录和助手中间态，自动拆分的状态门槛也比正式记录生命周期更窄。

**第三层检查（产品）**：
- 产品方向本身是合理的：用户应该只看到安静的记录与回顾，不应该接触系统内部的“材料不足”解释。
- 真正的问题不在产品目标，而在于底层记录资格与材料产线不统一。

**根本原因判断**：实现层为主，规范契约没有被统一落实。系统同时存在“正式 log 记录”“assist 中间 capture”“有 atoms 但无 payload 的历史路径”，而 Day Detail AI 只认 `atomization_payload.recordUnits`。
**建议修复方案**：
- 用共享 helper 定义正式记录资格；
- 让时间线 / Day Detail / 检索只读取正式 log 记录；
- 让正式 log 记录在 `.cleanReady` 时也能自动回到拆分队列；
- 用 Dev 诊断页暴露 `payload / recordUnits / state`，把排障从用户界面移到 DevTools。

## Handoff Notes
- `hasGeneratedNarrative` is intentionally hardcoded to `false` for now. Real persisted day narratives should be added in a separate pass instead of guessing from record count.
- The old `AI 轻回应` empty-state string was removed intentionally. If we want to debug missing AI materials later, that belongs in Dev tooling, not the user-facing day screen.
- Old assistant archive data may still show `atomsCount > 0` but `recordUnits == 0` until the passive or manual backfill runs. The new Dev diagnostic view now makes that visible and repairable.
- Historical formal records that came from the legacy assist-archive path can now be backfilled into `atomization_payload` without re-calling AI. The backfill derives `RecordUnitDraft` data from stored `AssistArchivePayload` units (or cleaned fallback text) and can run passively during review retrieval or manually from DevTools.
- `DayDetailScreen` was reorganized into calmer cards: `这一天` for quick context, `当天脉络` for the local/generated narrative, `原始片段` for chronological source reading, optional `AI 轻回应`, and `叙事引用来源` last. The goal is to let facts lead and AI follow.
- The new shared helper currently treats “formal record” as `mode == .log` (defaulting missing mode to `.log`). Hidden records still count as formal review material so they can continue participating in analysis.
- `Life Narattor.xcodeproj/project.pbxproj` contains a formatting-only diff that was already present in the working tree during verification and was not changed as part of this task.
- Weekly/monthly range review now uses current calendar week/month boundaries. If we later add a true `自定义` range review, it should reuse the same `RangeReviewData` builder instead of reintroducing a day-list-first layout.
