# 测试版上线前 Checklist

## 目标
这份清单用于当前测试版上线前的最后确认，覆盖：
- 核心功能是否可用
- 风险是否可控
- 测试用户如何免费使用现有 AI/API 能力
- 如何对测试用户设置可接受的配额限制，避免成本失控或被滥用

---

## 一、测试版范围确认

### 当前建议纳入测试版的功能
- 记录创建：文本、语音转写、记录详情查看
- 助手对话：文本输入、语音输入、整理为记录
- 助手记录草稿：全屏预览、轻量编辑、确认入库
- 拆分：记录单元拆分、结果预览、重新拆分
- 标签：显性标签库、隐性标签索引、隐性标签标准化 mapping
- AI 回顾：问题输入、事实与联系、证据折叠、继续追问
- 时间线：今天、7天回顾、30天回顾

### 当前建议继续隐藏或弱化的功能
- 旧的自然周/月自动回顾入口
- 仍不稳定的“空时间窗自动回顾”入口
- 任何尚未完成稳定性验证的批量自动整理入口
- TestFlight / Release 中的 Dev 工具入口；Debug 运行也默认隐藏，避免测试版截图或手动巡检误带出 Dev

### 上线前必须明确的产品边界
- AI 回顾是“问题驱动的回顾分析”，不是全量聊天助手
- 显性标签只来自用户主动创建或编辑；推荐标签系统当前关闭
- 隐性标签只服务索引、召回、回顾与提取，不直接暴露给测试用户

## 二、功能验收 Checklist

### 记录
- 可以创建文本记录
- 可以录音并完成转写
- 语音转写完成后整理文本正常显示
- 记录详情页可正常打开
- 重新转写、重新整理、重新拆分各自行为正确

### 助手
- 助手页文本对话正常
- 助手页语音输入正常
- 助手语音转写后会自动发回当前助手线程
- 点击“整理为记录”后会进入草稿编辑页
- 草稿确认后再入库，不会提前触发拆分和标签

### 拆分
- 拆分结果不再明显碎片化
- `summary / 上下文 / 过程 / 结果` 结构基本稳定
- 重新拆分时能显示清晰进度
- 拆分失败时有明确错误，而不是假装无网络

### 标签
- 隐性标签可生成、可在 Dev 查看
- 隐性标签标准化 mapping 可生成且不改原始数据

### AI 回顾
- 示例问题点击后会自动开始分析
- overview 查询能返回 `事实 / 联系 / 可继续问`
- focused 查询能返回分析、证据和相关记录
- 证据区默认折叠
- 继续追问能基于当前回顾材料继续问，而不是退化成普通聊天

## 三、稳定性与技术验收 Checklist

### App 基础稳定性
- `xcodebuild` 主工程构建通过
- Release/Archive 安装包底部不出现 Dev
- 关键页面切换无明显卡死
- 不出现主线程长时间阻塞
- 关键路径没有明显重复 AI 请求

### AI 链路稳定性
- backend proxy 正常
- AI 未配置时能明确报错，不返回假结果
- 主要 structured output schema 不再触发常见 HTTP400
- 标签建议与拆分能稳定返回结构化结果

### 数据安全
- 不在日志里记录敏感内容
- 不向测试用户暴露 OpenAI key / provider key
- 所有 AI 请求通过你自己的 backend proxy 统一转发

## 四、测试用户免费使用 API 的方案

## 核心原则
不要把 API key 发给测试用户。

测试用户的免费使用方式应该是：
- App 只请求你的 backend proxy
- backend proxy 再调用实际 AI provider
- 所有额度控制都放在你的 backend 上做

### 推荐做法
1. 每个测试用户有一个稳定标识
可选：
- 测试邀请码 + 本地生成的用户 ID
- 或者登录账号
- 或者安装后分配的 beta user id

2. backend 记录每个测试用户的 usage
最少记录：
- 日期
- 用户 ID
- 请求类型
- 请求次数
- 大致 token 数 / 音频时长
- 最后一次请求时间

3. 所有 AI 请求都先做额度检查
检查项目：
- 当日请求总数
- 当日音频转写时长
- 当日高成本请求数（如 AI 回顾、长文本整理）

4. 超额时返回明确错误
例如：
- `今日测试额度已用完，请明天再试`
而不是静默失败或返回奇怪空结果

## 五、测试期 API 限额建议

下面这套适合作为测试版第一版默认值，目标是：
- 对真实测试足够用
- 对滥用有明显约束
- 成本可控

### 按用户的日额度
- 文本/助手普通请求：`每日 80 次`
- AI 回顾请求：`每日 15 次`
- 语音转写：`每日 20 分钟`
- 助手整理为记录：`每日 20 次`
- 重新拆分/重新整理类重试：`每日 30 次`

### 单次请求上限
- 单条文本输入长度：例如 `不超过 6,000 字符`
- 单次 AI 回顾召回记录数：限制在安全范围内，例如 `50-100` 条候选后再压缩
- 单次语音长度：例如 `不超过 3 分钟`

### 更保守的高成本保护
- AI 回顾 follow-up 连续追问：每轮都算一次额度
- 长周期 narrative 生成：单独计数，单日限制更低
- Dev 工具入口不对测试用户开放

---

## 六、推荐的 backend 配额实现

### 最小可用版本
每次请求前：
1. 根据用户 ID + 日期读取 usage
2. 检查是否超额
3. 没超额则放行
4. 请求成功或失败后都记录一次 usage event

### 建议区分的请求类型
- `chat`
- `assist_archive`
- `atomize`
- `tag_suggest`
- `review_overview`
- `review_focused`
- `review_followup`
- `transcription`

### 推荐的控制粒度
- `rate limit`：防突发滥用
  - 例如每分钟不超过 `10` 次
- `daily quota`：防日级失控
- `cost bucket`：按高成本能力单独限额

---

## 七、如何让测试用户“免费但不失控”

### 最简单可执行方案
- 所有请求走你的 backend
- backend 识别测试用户
- 给每个测试用户固定的日额度
- 超额就停止服务并提示明日恢复

### 不建议的方案
- 直接把 provider API key 放进客户端
- 对测试用户完全不限量
- 用前端本地计数代替后端限制
- 只按请求数限制，不区分高成本与低成本请求

---

## 八、测试版前的后台最低能力

上线前建议至少具备：
- 测试用户身份标识
- usage 计数
- 日额度控制
- 高频率限流
- 错误日志
- 可人工封禁某个测试用户

如果时间允许，再补：
- 用户维度 usage 面板
- 每日成本概览
- 不同请求类型的比例统计

---

## 九、建议的测试版验收顺序

### 第一轮：功能能跑
- 记录
- 助手
- AI 回顾
- 标签建议

### 第二轮：数据链正确
- 拆分结果是否合理
- 标签是否复用显性标签库
- 隐性标签是否能生成并被标准化读取

### 第三轮：成本和限制
- 超额时能否正确拦截
- 不同请求类型是否被正确计数
- 高频调用是否会被限制

### 第四轮：真实测试用户体验
- 首次进入是否理解产品
- 主要入口是否清楚
- AI 回顾是否有价值
- 助手整理成记录是否顺手

---

## 十、当前最建议优先补齐的上线前事项
- 给 backend 增加测试用户 usage 计数与日额度限制
- 做一轮固定样本回归测试
- 清理剩余脏工作区并按主题提交
- 确认哪些 Dev 能力不会出现在测试版
- 为测试用户准备一版简短使用说明
