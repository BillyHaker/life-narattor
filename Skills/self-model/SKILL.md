# Self-Model（自我模型）

## 1. 设计原则

### 核心理念
「记录-进化」：Life Narrator 不只是日记，而是用户随时间演化的自我认知系统。
自我模型是这一理念的核心载体——它不是用户当下状态的快照，而是一条可追溯的进化时间线。

### 五条不可违背的原则

1. **只追加，不覆盖**：当期补丁队列只增不减；历史记录（含忽略条目）永远保留，不删除、不修改。
2. **用户永远是决策者**：AI 生成 Proposal，用户审批后才写入补丁；禁止自动修改自我模型。
3. **锚点神圣**：用户手写的基础锚点不受 AI 任何操作影响。
4. **格式面向未来**：补丁用结构化自然语言写，既人类可读，又可直接注入 LLM 上下文（无需转换）。
5. **提炼观察，不做评判**：系统从用户的行为记录和所思所想中提炼模式，不判断行为好坏；「发现规律」是任务边界，「评判对错」不是。

## 2. SelfModelEntity 数据结构

自我模型由四个部分组成，各部分职责不同，更新频率不同：

```
┌─────────────────────────────────────────────────┐
│  锚点          用户手写，永不被 AI 修改           │
├─────────────────────────────────────────────────┤
│  主体          每三个月整合一次，由当期补丁重写   │
├─────────────────────────────────────────────────┤
│  当期补丁      距上次整合后的增量，随时追加       │
├─────────────────────────────────────────────────┤
│  忽略条目      用户要求忽略内容的集中化，进入     │
│                Prompt 作为负向指令               │
└─────────────────────────────────────────────────┘
```

### 2.1 锚点（用户手写，不可被 AI 覆盖）

```json
{
  "anchor_id": "anchor_001",
  "created_at": "2026-03-08T10:00:00Z",
  "content": "我是一个做事比较慢热但想清楚了会很专注的人，对健康和长期主义比较在意。",
  "editable_by": "user_only"
}
```

- 用户可自行编辑，每次编辑生成新版本（旧版本保留）。
- AI 读取但不写入。

### 2.2 主体（季度整合，由 ConsolidationJob 重写）

主体是当前自我模型的干净版本，每三个月由 ConsolidationJob 从当期所有 active 补丁归并重写。

```json
{
  "body_id": "body_2026Q1",
  "consolidated_at": "2026-03-31T02:00:00Z",
  "period": "2026-01 ~ 2026-03",
  "content": "[2026 Q1 自我模型主体]\n\n行为规律：运动习惯在工作日中段最稳定...\n当前关注点：Life Narrator 产品设计、个人知识管理...\n沟通偏好：倾向先想清楚再表达，书面优于口头...",
  "source_patch_ids": ["patch_001", "patch_002", "patch_005"]
}
```

- 每次整合后旧主体归档（不删除），新主体替换为当前版本。
- 当期补丁队列在整合后清空，重新开始积累。

### 2.3 当期补丁（距上次整合后的增量）

每条补丁追加存储，季度整合后队列清零：

```json
{
  "patch_id": "patch_20260308_001",
  "created_at": "2026-03-08T08:00:00Z",
  "dimension": "behavioral",
  "trigger": "weekly_job",
  "status": "active",
  "content": "[2026-03-08 行为层补丁]\n观察到：过去四周运动相关记录共 12 条，坚持晨跑的描述集中在周二至周四。\n新增认知：运动习惯在工作日中段最稳定，周末反而容易中断。\n证据引用：Atom#334, Atom#389, Atom#401",
  "approved_at": "2026-03-08T09:30:00Z",
  "approved_by": "user"
}
```

**status 枚举**：
- `pending`：等待用户审批
- `active`：已批准，参与主体构建和 Prompt 注入
- `dismissed`：用户驳回，不写入补丁（ProposalEntity 层处理）

### 2.4 忽略条目（集中化负向指令，进入 Prompt）

忽略条目不是简单的「跳过某条补丁」，而是将用户所有忽略请求集中整理成一份清单，在 Prompt 中以负向指令形式注入，确保 AI 不会从新数据中重新发现并呈现用户已明确不想看到的内容。

```json
{
  "ignore_list_id": "ignore_2026Q1",
  "updated_at": "2026-03-08T10:00:00Z",
  "entries": [
    {
      "entry_id": "ign_001",
      "added_at": "2026-02-15T00:00:00Z",
      "description": "关于「周末运动中断」的规律，用户已知晓，不需要再提及",
      "source_patch_id": "patch_20260115_003"
    },
    {
      "entry_id": "ign_002",
      "added_at": "2026-03-01T00:00:00Z",
      "description": "关于「压力下回避决策」的描述，用户认为已不准确，请忽略",
      "source_patch_id": "patch_20260201_007"
    }
  ]
}
```

**在 Prompt 中的注入形式**：
```
[忽略条目 - 请勿重新提及以下内容]
- 关于「周末运动中断」的规律，用户已知晓
- 关于「压力下回避决策」的描述，用户认为已不准确
```

季度整合时，ConsolidationJob 会合并重复/过时的忽略条目，保持列表简洁。

### 2.5 六个维度

补丁和主体内容按六个维度组织：

| 维度 key | 中文名 | 更新频率 | 说明 |
|---------|-------|---------|-----|
| `values` | 核心价值观 | 月度 | 最稳定，需要强证据支撑才更新 |
| `thinking_patterns` | 思维模式 | 月度 | 决策风格、认知偏好 |
| `current_context` | 当前上下文 | 周度 | 当前项目、关注点（最易变） |
| `behavioral_patterns` | 行为规律 | 周度 | 习惯、时间规律、行动倾向 |
| `communication_prefs` | 沟通偏好 | 月度 | 语言风格分析结果 |
| `evolution_tracking` | 进化追踪 | 自动 | 各维度变化的 delta 历史索引 |

## 3. 周期更新 Job

### 3.1 三种触发类型

#### BehavioralUpdateJob（每周一凌晨，后台静默执行）

```
输入：最近 7 天的 Atoms（按时间顺序）
过滤：优先取行为/情绪维度标签（confidence ≥ 0.65）的 Atoms
分析：
  - 高频动词/形容词统计
  - 时间规律检测（哪些行为集中在哪些时段）
  - 与当前 behavioral_patterns 层补丁比对
输出：Proposal（若无显著变化，不生成 Proposal，静默结束）
触发通知：仅在 Proposal 置信度 ≥ 0.7 时推送通知
```

#### IdentityUpdateJob（每月 1 日，后台静默执行）

```
输入：最近 30 天的 Atoms
分析：
  - 重复出现的主题/关注点聚类
  - 否定信号检测（「不想」「放弃」「意识到」「以前认为」）
  - 语言模式分析（句式、情绪基调）
  - 与 values / thinking_patterns / communication_prefs 层比对
输出：Proposal（分维度，每个维度独立一条 Proposal）
触发通知：仅在至少一条 Proposal 置信度 ≥ 0.65 时推送
```

#### ContradictionDetector（每次 Atom 写入后触发，轻量）

```
输入：新写入的 Atom + 当前 active 补丁摘要（仅取 values/thinking_patterns 层）
分析：快速比对，检测明显矛盾
阈值：矛盾分数 > 0.8 → 标记为矛盾事件
累积：同一维度累计 3 个矛盾事件 → 触发小型 IdentityUpdateJob
```

#### ConsolidationJob（每季度首日，后台静默执行）

```
触发时间：每年 1/4/7/10 月 1 日凌晨

输入：
  - 当期所有 status=active 的补丁
  - 当前忽略条目列表
  - 上一版主体（作为参考）

执行步骤：
  1. 将当期所有 active 补丁按六维度归并，重写主体内容
  2. 合并去重忽略条目（删除过时/重复条目，保留清单简洁）
  3. 将旧主体归档（body_id 加 _archived 后缀，保留不删）
  4. 新主体写入，当期补丁队列清空

输出：
  - 新版主体（body_id = body_YYYY_QN）
  - 更新后的忽略条目列表
  - 归档记录写入 Session Log

约束：
  - 整合过程不生成 Proposal，不需要用户审批（是合并，非创作）
  - 若 active 补丁数量为 0，跳过主体重写，仅更新忽略条目
  - 整合失败时保留旧主体，写入错误日志，下次启动重试
```

### 3.2 Job 执行约束

- 所有 Job 在后台线程执行，不阻塞 UI。
- 网络不可用时跳过，下次启动补偿执行。
- 单次 Job 最多消耗 500 个 Atoms（超出则取最近 500 条）。
- Job 执行结果写入 Session Log（`Docs/04_Sessions/`）。

## 4. Proposal（用户审批系统）

### 4.1 ProposalEntity 结构

```json
{
  "proposal_id": "prop_20260308_001",
  "created_at": "2026-03-08T08:00:00Z",
  "source_job": "BehavioralUpdateJob",
  "dimension": "behavioral_patterns",
  "confidence": 0.82,
  "proposed_patch": {
    "dimension": "behavioral",
    "content": "[2026-03-08 行为层补丁]\n观察到：...\n新增认知：...\n证据引用：Atom#334"
  },
  "evidence_atom_ids": ["atom_334", "atom_389", "atom_401"],
  "status": "pending",
  "user_action": null,
  "user_action_at": null
}
```

### 4.2 用户审批操作

| 操作 | 结果 |
|-----|-----|
| 接受 | Proposal → Patch（status=active）；ProposalEntity status=accepted |
| 修改后接受 | 用户编辑 content → Patch（status=active，注明 user_edited=true）|
| 忽略 | ProposalEntity status=dismissed；本条不写入补丁 |
| 忽略并反馈 | status=dismissed；AI 在下次 Job 时参考 dismiss 原因 |

### 4.3 审批界面要求

- 入口：通知 → 「查看自我模型更新建议」
- 展示：证据 Atoms 可展开查看原文
- 操作：接受 / 修改 / 忽略（三个按钮，忽略需二次确认）
- 状态：未读 badge 显示待审批数量

## 5. 第二自我对话引擎

### 5.1 Prompt 构建契约

```
System Prompt =
  [锚点]              ← 置顶，永远包含
  + [主体]            ← 最新季度整合版本
  + [当期补丁]        ← 距上次整合的增量补丁（按时间顺序）
  + [忽略条目]        ← 负向指令：「以下内容请不要重新提及…」
  + [相关 Atoms]      ← 按查询意图检索，Top-K
  + [模式标识]        ← 见下方两种模式定义
```

**Token 预算分配（参考）**：

| 部分 | Token 估算 | 说明 |
|-----|-----------|-----|
| 锚点 | ~200 | 固定，用户手写 |
| 主体 | ~800 | 季度整合版，随时间缓慢增长 |
| 当期补丁 | ~不限（当前上下文已足够）| 最多 3 个月增量，通常 < 2000 token |
| 忽略条目 | ~200 | 精简列表，季度整合后去重 |
| 相关 Atoms | ~300 | Top-5 |
| 模式指令 | ~100 | 固定模板 |

> **架构原则**：四层结构均采用结构化自然语言，设计上支持全量直接注入。季度整合控制了主体和补丁的总体积；忽略条目的存在确保负向约束不会随季度清空而丢失。

### 5.2 模式 A：主题回顾

```
[模式标识]
你是一个记忆检索助手，帮助用户回顾特定主题的经历。
以第三人称客观视角整理以下记录，不做评判，不做建议。
```

**检索策略**：
- 输入关键词 → 标签匹配（confidence ≥ 0.65）→ 时间排序
- 返回 Top-5 相关 Atoms 作为上下文

**输出格式**：时间线 + 关键片段引用 + 可选「晶体卡片」

### 5.3 模式 B：第二自我模拟

```
[模式标识]
你就是用户本人。以下是你的自我认知和近期经历。
请以第一人称，用你自己的语气和思维方式回答问题。
不要扮演助手，不要提建议，说的是「你自己」会怎么想、怎么做。
```

**检索策略**：
- 输入问题 → 意图分类（轻量模型）→ 多维标签检索 → 相关度排序取 Top-5

**输出格式**：自然对话，语气与用户历史记录中的语言风格一致

### 5.4 晶体卡片格式（Crystal Card）

主题回顾的结构化输出单元：

```json
{
  "card_type": "crystal",
  "title": "关于运动习惯的认知演化",
  "time_range": "2026-01 ~ 2026-03",
  "core_insight": "运动对我来说不是目标达成，而是状态管理的工具。",
  "evidence": ["Atom#334（2026-01-15）", "Atom#389（2026-02-03）"],
  "evolution_note": "从「坚持打卡」心态 → 「按状态灵活调整」心态",
  "confidence": 0.85
}
```

## 6. 数据存储与隐私

- SelfModelEntity 和 PatchEntity 存储于本地 CoreData，遵循 `database-schema` 规范。
- 任何补丁内容在发送至 AI 前，必须经过 `privacy-redaction-standard` 脱敏处理。
- 用户可在设置中导出或删除全部自我模型数据。
- 云同步（如有）需加密传输，不得以明文存储于服务端。

---

## 7. 验收标准

- [ ] AI 无法绕过 Proposal 系统直接写入补丁
- [ ] 用户锚点在任何 Job 执行后保持不变
- [ ] Prompt 构建包含忽略条目作为负向指令
- [ ] ConsolidationJob 执行后旧主体归档、补丁队列清空、忽略条目去重
- [ ] 主题回顾和第二自我模拟可通过模式参数切换
- [ ] 晶体卡片输出包含 evidence 引用，可追溯至原始 Atom
- [ ] 所有 Job 执行记录写入 Session Log
- [ ] 用户可查看、编辑、忽略任意 Proposal

## 8. Backlog（暂不实现，保留方案）

### 冷启动加速方案

新用户前三个月数据稀少，第二自我体验较弱。以下方案已设计，待用户测试反馈后决定是否实现：

- **方案 A：引导问卷**：首次进入第二自我时，提供一套结构化问题（10–15 题），覆盖六个维度，引导用户快速建立初始锚点和主体草稿。
- **方案 B：对话式建档**：通过多轮对话（类似 Assist 模式）逐步提炼用户的自我认知，完成后自动生成初始主体。
- **方案 C：导入已有内容**：允许用户导入其他日记/笔记应用的历史数据，作为冷启动素材进行分析。

> 当前 V1 不实现上述方案，仅依赖用户手写锚点作为冷启动内容。
